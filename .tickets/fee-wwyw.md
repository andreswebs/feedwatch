# schema command

Implement the schema command: generate a machine-readable JSON contract by introspecting the live urfave command tree (args, flags, types, defaults) plus a per-command registry for exit codes and output JSON Schema, with schema COMMAND narrowing to one. Refs: docs/cli-design.md (Discoverability, CLI Framework); docs/research/urfave-cli.reference.md.

## Design

The `schema` command: emit a machine-readable description of the interface,
generated by introspecting the live urfave command tree plus a per-command
registry for the parts urfave does not track.

```go
&cli.Command{ Name:"schema", ArgsUsage:"[COMMAND]",
  Arguments: []cli.Argument{ &cli.StringArg{Name:"command"} }, Action: schemaAction }

type CommandSchema struct {
  Command   string            `json:"command"`
  Args      []ArgSchema       `json:"args"`
  Flags     []FlagSchema      `json:"flags"`
  ExitCodes map[string]string `json:"exit_codes"`
  Output    json.RawMessage   `json:"output_schema"`
}
```

- Walk `cmd.Root().Commands`; for each, read `Flags` (name, type, default via a
  type switch over the concrete `*cli.XxxFlag`) and `Arguments`.
- A per-command registry (a map keyed by command name) supplies the exit-code
  table and the output JSON Schema, which urfave does not know. Keeping it beside
  the command definitions ensures `schema` cannot drift from the real flags.
- `schema` with no arg emits all commands; `schema <command>` narrows to one.
- urfave ref: cli.Command fields (Commands, Flags, Arguments), FlagNames; the
  concrete flag types (StringFlag/BoolFlag/IntFlag/DurationFlag/StringSliceFlag).

TDD plan (cmd.Run capturing stdout):

1. (tracer) `schema poll` emits JSON with poll's flags (names + types) and its
   exit codes (0/2/3) and output schema.
2. `schema` (no arg) lists every registered command.
3. flag types are reported correctly (bool/string/int/duration/stringSlice).
4. an unknown command name -> usage error, exit 1.
5. the introspected flag set matches the command's actual flags (add a flag in a
   test command -> it appears) — guards against drift.

Deep-module note: `schema` reads the command tree it is part of; the registry
holds only what introspection cannot derive.

## Acceptance Criteria

- `schema` introspects the urfave command tree for args/flags/types/defaults and
  augments with a registry for exit codes + output JSON Schema; `schema
<command>` narrows; unknown command -> exit 1.
- Behaviors 1-5 covered; drift guard included.
- Supports Req 4. `make validate` passes.

## Notes

**2026-06-30T00:16:35Z**

Implemented schema command. internal/cli/schema.go introspects the live urfave tree (cmd.Root().Commands) for args/flags/types/defaults; schema_registry.go holds the per-command exit codes + output JSON Schema that introspection cannot derive. Bare 'schema' emits {commands:[...],global_flags:[...]}; `'schema <command>'` narrows to a CommandSchema; unknown command -> CatUsage/ErrUsage exit 1 (reuses root.go unknownCommandErr).

Key decisions: (1) Flag type via a TYPE SWITCH over concrete *cliv3.XxxFlag (not TypeName()), because TypeName() reports a StringSliceFlag as 'string' (it returns the slice ELEMENT type) — the switch yields 'stringSlice' correctly. Default-value omitted when zero (matches the design's --force example with no default). Fallback for unknown future flag types uses the DocGenerationFlag.TypeName() interface (TypeName is NOT on the base Flag interface). (2) Args introspected via type switch too — the Argument interface exposes no Name accessor, only HasName;*StringArg vs *StringArgs distinguishes singular/variadic. (3) Filter out the auto-added 'help' command and the --help/--version flags: the design treats --help/--version separately from the machine-readable contract. (4) Hidden commands (completion helper) skipped. (5) Drift guard test appends a command with a flag to a live root and asserts it appears in its schema with no registry change.

Note: poll declares no Arguments (it uses cmd.Args().Slice()), so its args list is empty — schema reflects the real tree, not the docs/cli-design.md aspirational {name:feed,variadic:true} example. If feed-arg discoverability is wanted, add Arguments:[]cliv3.Argument{&cliv3.StringArgs{Name:"feed"}} to pollCommand. Registry output schemas are object/top-level-property granularity (not every Item field) to bound hand-maintenance drift; export's schema notes it emits OPML XML, not JSON.
